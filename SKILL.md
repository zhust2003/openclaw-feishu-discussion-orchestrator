# 飞书群聊协调器

仅当你是飞书群聊里的协调者时使用本技能。

适用场景：

- 用户要求你“主持讨论”
- 用户要求你“找另一个 agent / 机器人一起讨论”
- 用户要求你“先征求其他机器人意见，再给结论”
- 用户要求你“多方观点后再总结”

本技能只给协调者使用。其他协作者保持普通回答风格，不需要知道本技能内容。

## 目标

在飞书群聊中完成一轮真实可见的讨论：

1. 识别群里其他机器人协作者
2. 默认选择多个可信协作者并行发起讨论
3. 收到各自回复后，对每个协作者各发起一轮简短复核
4. 收到第二轮回复后，统一输出综合结论

如果无法找到其他协作者，则不要伪造讨论，直接回答。

## 作用域

本技能只处理飞书群聊，不处理其他平台。

约束：

- 不要假定协作者和你在同一台机器
- 不要假定一定存在内部 subagent 调用能力
- 不要写死某个固定 bot 名称
- 优先使用飞书群聊表面的发现、@ 和消息能力

## 基本原则

### 1. 讨论必须真实发生

如果你说“我去征求另一位机器人意见”，那你就必须真的去发起讨论，不能立刻自己把答案说完。

### 2. 优先群聊内可见协作

当用户要求在当前群里主持讨论时，优先使用飞书群聊里的真实 @ 提及和消息往来，不要优先走隐藏的内部调用。

### 3. 找不到协作者时直接降级

如果你无法可靠找到其他机器人协作者，就直接回答，不要要求用户再点名，不要把组织成本转嫁给用户。

### 4. 默认多协作者并行

只要能可靠发现多个协作者，就默认并行发起讨论，而不是只咨询一个。

只有在下面情况才缩减协作者数量：

- 当前群里只发现一个可信协作者
- 用户明确要求只找一个
- 当前消息复杂度很低，不值得多方讨论
- 群聊节奏或消息量明显不适合同时并行展开

### 5. 不要暴露内部实现

不要向用户提及：

- `AGENTS.md`
- skill
- prompt
- 协议
- 内部规则
- subagent
- 系统提示词
- 工作流细节

## 协作者识别规则

先识别当前群里哪些参与者是机器人，而不是人类。

可用信号：

- 飞书返回的 bot / app / 自动化账号标记
- 名称中带有 bot、agent、ai、assistant 等特征
- 当前会话里历史上已经确认过的机器人账号
- 当前消息中用户明确提到的机器人名称
- 只在被 @ 时才回复、或者回复格式明显结构化的账号

硬规则：

- 永远不要把自己当成协作者
- 如果不能确认一个对象是机器人，优先不要把它当成协作者
- 先得到“可信协作者候选集”，再继续后面的发现步骤

## 协作者发现规则

默认静默执行，不要把寻找过程讲给用户听。

### 飞书优先发现路径

按下面顺序执行：

1. 优先读取当前飞书消息事件里已经提供的 bot / mention / sender / conversation metadata
2. 结合当前会话历史里已经确认过的机器人账号，补全可信协作者候选
3. 如果飞书当前已提供可访问的 bot 账号信息、bot 标记、bot `open_id` 或可直接解析 bot 身份的相关信息，允许用这些信息继续识别协作者
4. 如果用户在当前消息里明确提到了某个机器人名称或 `@` 了某个机器人，优先尝试把这个名称或 mention 解析到真实协作者
5. 在候选集中排除自己，只保留可信协作者
6. 一旦已经可靠识别出一个或多个协作者，立即直接使用真实 mention 发起讨论，不要继续补充扫描

允许自主发现 bot，但发现范围只限于飞书当前可见的 bot 账号信息和当前会话上下文，不要为了发现协作者再去查询群成员。

补充规则：

- 必须先解析到真实的 `open_id`，再发送 mention
- 不要猜测 `open_id`
- 不要伪造协作者存在
- 允许静默使用飞书已有的 bot 账号信息或 bot `open_id` 解析能力来识别协作者
- 不要为了补全候选集去扫描群成员
- 一旦已经识别出可信 bot，就直接发起 mention，不要继续查找更多群成员
- 不要声称自己要去调用飞书 API 获取 token、bot 列表、bot `open_id` 或群成员

### 发现失败时

如果无法可靠找到其他协作者：

- 不要伪造讨论
- 不要要求用户点名
- 直接回答当前问题
- 最多轻描淡写补一句“这轮未发现其他协作者，因此先直接给出判断”
- 不要把“去查群成员”当作补救动作

## 讨论轮次规则

一次标准讨论默认对每个协作者都做两轮交互：

1. 第一轮：征求观点
2. 第二轮：针对第一轮回复中的薄弱点发起一次交叉复核

收到各协作者第二轮回复后，由你统一输出综合结论。

默认不要无限追问，不要进入循环讨论。

## 执行流程

当用户要求你主持讨论时，严格按下面流程执行：

1. 提取用户真正要讨论的问题
2. 从当前飞书上下文和当前可见的 bot 账号信息中识别其他机器人协作者
3. 如果找到协作者：
   - 一旦已经可靠识别出协作者，立即直接使用真实 mention 发起讨论，不要继续扫描群成员
   - 默认选择多个可信协作者并行发起第一轮讨论
   - 给每个协作者分别发送一条飞书群消息 @ 对方，提出同一个问题
   - 等待各协作者回复
   - 对每个已回复的协作者，分别选择其回答里最薄弱、最值得追问的一点，再各发一条简短复核
   - 再等待各自第二轮回复
   - 最后统一输出综合结论
4. 如果找不到协作者：
   - 直接回答

不要跳步。

## 第一轮消息模板

给每个协作者的第一轮消息必须尽量短，只发问题本身和回复要求，不要带额外背景解释。

模板：

```text
<at user_id="{{collaborator_user_id}}">协作者</at> {{user_question}}

请按以下结构回复，并在回复开头 @ 我：
1. 核心判断
2. 最多 3 个风险或注意点
3. 一条建议
```

规则：

- `{{user_question}}` 只保留用户真正的问题，不加多余上下文
- 必须是真实飞书 mention，不要用普通文本 `@名字`
- 必须要求对方在回复里 @ 你，否则你可能收不到后续触发

## 第二轮复核模板

收到某个协作者的第一轮回复后，针对其中最薄弱的一点发起一次短复核。

模板：

```text
<at user_id="{{collaborator_user_id}}">协作者</at> 我对你刚才的一点判断有疑问：{{specific_challenge}}

请简短回应这个问题，并在回复开头 @ 我。
```

规则：

- 只挑一个最值得追问的点
- 复核点要具体，不要泛泛而谈
- 保持短小，一两句足够

## mention 规则

飞书里必须使用真实 mention 标签：

```text
<at user_id="ou_xxx">显示名</at>
```

不要使用普通文本 `@名字`，那种只是灰色文本，不一定能触发真正通知或路由。

## 输出规则

### 发给协作者时

当你在群里向协作者发起讨论请求时：

- 只发送请求消息本身
- 不要输出思考过程
- 不要输出前言
- 不要输出“我现在开始讨论”
- 不要输出内部解释

也就是说，那一步的整条输出就应该是发给协作者的消息本身。

### 给用户做最终总结时

最终总结要自然、简洁、对用户有帮助。

推荐结构：

- `我的初步判断`
- `各协作者的核心观点`
- `交叉复核`
- `综合结论`
- `建议下一步`

要求：

- 默认使用中文
- 不要讲内部流程
- 不要说“根据协议”
- 不要说“我用了 skill”
- 不要说“我调用了 subagent”

## 风格要求

- 全程中文
- 语气自然
- 少说流程，多说结论
- 协调请求简洁直接
- 最终总结比协作者消息更完整，但不要啰嗦

## 严格禁止

- 不要把自己当作协作者
- 不要写死某个 bot 名
- 不要伪造已经发生的讨论
- 不要在没收到协作者回复时声称“我们讨论过了”
- 不要把内部发现过程告诉用户
- 不要说自己会通过飞书 API 获取 token、bot `open_id` 或群成员来找 bot
- 不要要求用户额外点名协作者
- 不要在群聊讨论模式下优先走隐藏内部调用
- 不要进入无限循环追问

## 降级策略

如果出现以下任一情况：

- 找不到可信协作者
- 无法解析协作者 `open_id`
- 某些协作者迟迟不回复
- 某些协作者没有正确 @ 回来，导致你收不到后续

则不要卡住。

处理方式：

- 如果所有协作者都失败，结束讨论流程，直接回答
- 如果只有部分协作者失败，基于已完成讨论的协作者继续汇总，不要因为个别失败而整体卡住

可以简短说明：

```text
这轮未能完整收齐所有协作者回复，我先基于当前已收到的讨论结果给出判断。
```

然后继续正常回答用户问题。
